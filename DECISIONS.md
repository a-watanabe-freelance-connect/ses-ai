## 1. 基本方針
- **目的**: ① 案件メール取込（→ `案件台帳`）／② 要員→案件マッチ（本人タブに蓄積）。
- **方式**: ハイブリッド。運ぶ・絞るのはスクリプト（**Python**）、読む・判断は Claude。書込/読取もスクリプト、対話中の軽い読取のみ Sheets MCP を補助。
- **運用**: 当面は手動（Claude Code 都度実行）。無人化（cron）は §6 の解禁条件が揃うまで禁止。
- **可搬性**: 絶対パスに依存しない。スクリプトはリポジトリルート相対で `secrets/`・`data/`・`.env` を解決する（誰の環境・どのマシンでも動く）。

## 2. スコープ
- **v0（今つくる最小スライス）**: 案件取込＋要員→案件マッチが端から端まで一周する最小。
- **backlog（v0 が動いてから足す堅牢化）**: §6。消さず「後で」。
- **将来（別軸・当面作らない）**: 人材取込（`人材台帳`）・案件→人材（方向B）・添付スキルシート抽出・機械フィルタ `classify`。

## 3. 規模と取込方針
- **1日 約5,455通（案件メールは概算約1,850通/日）/ 約1,500社**（2026-07-02実測。`MILESTONE0_REPORT.md`。当初想定「1日約1,000通」から約5.5倍の乖離・レビュー2026-07-06で指摘）。各社が独自配信で**案件メールの重複は多くない見込み**（人材メールでは重複配信を確認済み・MVP対象外）。
- **SES 専用受信箱＝ノイズ極少（ゼロではない）** → キーワード機械フィルタは作らず、**案件/人材/その他の仕分けは Claude の推論**。純ノイズ配信元（ニュースレター等）のみ送信元ドメイン単位の軽量スキップリスト（`fetch_gmail.py` の `SKIP_BODY_DOMAINS`）で本文全文読みを抑止（メールは残しラベル付与＝取りこぼしなし）。
- **MCP でのメール取込は不採用**（数千通規模の生本文をコンテキストに乗せられない）。取得はスクリプト、構造化は絞った候補だけ Claude。
- 規模が上がった時の節約レバーは**フィルタでなく** 既定 Haiku＋バッチ化＋キルスイッチ。手動50〜100通/バッチは実測規模では55〜110バッチ/日相当となり運用が破綻するため、無人化（Message Batches API＋Haiku）をv0完走後すぐの優先課題とする。

## 4. v0 の作り
- **取込**: Gmail を `-label:SES-AI/processed` で未処理取得 → Claude が案件のみ構造化 → `案件台帳` に **`messageId` 列で重複回避**して追記 → 各メールに `SES-AI/processed` ラベル。**カーソルは Gmail ラベルだけ**（時刻カーソルを使わない＝取りこぼし無し）。
- **マッチ**: `案件台帳` を全件読む → Claude が許容条件（隣県可・経験± 等）込みで判断 → 本人タブに **1行1案件**（本人タブの `messageId` で重複回避）。
- **本人条件の置き場所を確定 → md ファイル**: 自社要員のプロフィール（スキル/経験/希望県/単価/リモート/稼働開始）は `要員/<名前>.md` で人間が用意・維持し、マッチ時に Claude が読む。**マッチ結果は従来どおりスプレッドシートの人員タブ（1行1案件）**。「人員タブ上部」「人員一覧に列」「都度プロンプト」案は不採用。
- **複数人運用**: 共有カーソルは重い `_state` でなく **Gmail ラベル（共有メールボックス側）＋シートの `messageId` 列**で足りる。ロック/カーソル/実行履歴は backlog。
- **スキーマ**: Claude 出力を `案件台帳` の主要列へ。`row_key=messageId`。**★＝マッチの核**（スキル/経験/単価/県/リモート/開始/商流）。抽出不能は空欄（推測で埋めない）。単価は上限予算（min/max は backlog）。

## 5. 50体レビューの結論（2026-07-01 実施）
- 判定「**条件付き前進可**」。骨格（コード/Claude 境界・★対称スキーマ・空欄非補完・段ごと冪等な中間ファイル）は高評価＝**維持**。実装の不変条件を backlog（§6）に確定。
- **争点の裁定**: Sheets は当面継続（数値トリガでDB移行）／名寄せは案件と人材で非対称（案件=候補フラグ・自動統合しない／人材=既定オフ）／人員タブは 1人1タブ（`person_id` 化は backlog）／構造化の既定は **Haiku**（Opus は★欠落再抽出・マッチ限定）。

### 6-1. cron化フェーズ1: `_state`/`_runlog`基盤（実装済み・2026-07-06）
- **実装**: `scripts/state.py`（新規）。`_state`（key-value 4列: `key, value, updated_at, updated_by`）と `_runlog`（append-only 10列。2フェーズ＝開始行＋終了行をrun_idで突合）をSheetsタブとして追加。`fetch_gmail.py`・`append_sheet.py`・`setup_v0a.py`（`ensure_state_tabs`・冪等）に統合。
- **確定値**: `daily_budget=8000`（セーフティネット方式。実測5,455通/日・案件約1,850通/日を明確に上回る値を採用。業務上の絞り込みではなく異常時のみ発動する安全弁として運用する）。`lock_ttl_sec=21600`（6時間）。`kill_switch`は独立フラグ（既定`off`）。
- **ロックの設計判断**: 本命のレースは「fetch実行中の衝突」ではなく「fetch後に人が数時間かけてtriage/structure作業をしている間に、別の人が追いfetchして同じ未処理メールを重複取得すること」（ラベル付与はappend_sheet.py最終段でしか起きないため）。よって**ロックはスクリプト単体でなくセッション単位**（`fetch_gmail.py`が取得し`append_sheet.py`完了時に解放）とした。TTLはプロセス異常終了時のフォールバック上限であり、通常の保持時間はこれより短い想定。
- **予算消費はfetch側だけで行う**: コスト実体はtriage/structure段階（Claude推論）で発生し、appendは後始末に過ぎないため、二重管理を避けappend_sheet.pyでは予算チェックしない。
- 手動復旧用CLI: `python scripts/state.py lock status|release [--force]` / `budget status` / `killswitch on|off|status`。
- 残る項目（今回のスコープ外）: overlap 再取得／`_staging` 退避／`row_key=messageId#内容ハッシュ`によるupsert・コミット順序固定。下記2点として引き続きbacklog。

- **カーソル state 化（残り）**: overlap 再取得／`_staging` 退避。※ ロック・予算上限・キルスイッチ・監査ログは6-1で実装済み。

## 7. Skill化と無人化は別軸（2026-07-03決定）
- **Skill化**（`.claude/skills/` 配下の各スキル。構成は§8）は、有人実行の手順・判断基準を`SKILL.md`に固定するもの。§6の無人化解禁条件（キルスイッチ・監査・鮮度・ゴールドセット等）を**待たずに着手してよい**。理由: 有人実行では異常発生時に人間がその場で気づき修正できるため、無人化特有のリスク（暴走コスト・エラー放置）が生じない。
- **無人化**（cronでの自動実行）は引き続き§6の条件が揃うまで解禁しない。Skill化された手順をそのまま無人トリガーに繋ぐ形を想定。

## 8. スキルは3構成（2026-07-03決定）
- Skill は **`/ses-triage`（案件/人材メールの仕分け）・`/ses-structure`（案件メールの構造化→案件台帳追記→ラベル付与）・`/ses-match`（要員→案件マッチング）** の3構成とする。旧計画の `/ses-import`（仕分け＋構造化の一体スキル）は前2者に**分割し廃止**。
- 分割理由: ①1日1,000通規模のバッチを段単位で中断・再開できる ②仕分け精度と構造化精度を別々に点検できる（ゴールドセット回帰も段別に組める） ③将来 `classify`（Haiku一次分類）導入時に仕分け段の差し替えだけで済む。
- 中間データ契約を追加: **`data/triage/*.jsonl`**（1行= `{messageId, 分類: 案件|人材|その他, 理由}`。ファイル名は対応する inbox と同名・全 messageId を網羅）。
- 各スキルの**正本仕様は `docs/skills/ses-triage.md`・`ses-structure.md`・`ses-match.md`**（2026-07-03作成）。`.claude/skills/*/SKILL.md` は仕様書に準拠して実装・更新する（仕様変更は仕様書を先に直す）。
- 運用上の不変条件: `SES-AI/processed` ラベル付与は `/ses-structure` 最終段（`append_sheet.py`）のみが行う。仕分けだけで止めてもメールは未処理のまま残る（取りこぼし無し）。逆に **append 前には inbox 全件の仕分け完了・案件全件の構造化完了を突合確認する**（未了のままラベルされるとそのメールは以後取得されない）。
