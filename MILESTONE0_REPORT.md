# マイルストン0 実測レポート（2026-07-02実施）

> TODO.md「マイルストン0: まず実測」に対応する実測記録。ここは**生の実測結果と分析**を残す場所で、これを踏まえた確定事項は `DECISIONS.md`、残作業は `TODO.md` に反映する（本レポート単体では方針は変えない）。

## 1. 実行内容

1. `scripts/fetch_gmail.py --date 2026-07-01` で7/1宛の未処理メールを取得（`after:`/`before:`クエリ使用）
2. 取得結果を `received_at`（JST）で7/1カレンダー日に厳密フィルタ
3. フィルタ後の集合から80件をランダム抽出（`random.seed(42)`）
4. 80件をサブエージェントが1件ずつ読み、案件/人材/その他に仕分け→案件のみ`案件台帳`スキーマへ構造化
5. `scripts/append_sheet.py` で案件台帳へ追記＋対象82件（サンプル80件＋事前テスト2件）に`SES-AI/processed`ラベル付与

## 2. 取得件数の齟齬について（5,970件 → 5,455件）

`--date 2026-07-01` で取得したメールは **5,970件**だったが、`received_at` の実際の範囲は
`2026-06-29T16:00:45+09:00` 〜 `2026-07-02T08:59:39+09:00`（約2.7日分）に広がっていた。

**原因**: Gmail検索の `after:`/`before:` 演算子は日付のみを指定する仕様で、JSTのカレンダー日と厳密には一致しない（Gmail側のタイムゾーン解釈・検索インデックスの境界の丸めによるものと推測される）。今回は指定日の前後にはみ出す形で、本来の範囲より広く取得された。

**実害の程度**: `received_at` が `2026-07-01` で始まるものだけに絞ると **5,455件**（全体の91.4%）。はみ出し分は515件（8.6%）で、量的な結論（後述）を左右するほどの誤差ではない。

**対応**: 今回はローカルで`received_at`文字列フィルタをかけて7/1のみに絞り込んでから分析・サンプリングした。`fetch_gmail.py`の`--date`オプション自体は暫定実装（Gmail検索頼み）のままなので、日次バッチ等で日付境界の精度が必要になった場合は要改修（例: 取得後にreceived_atでのポストフィルタをスクリプト側に組み込む）。

### 3.4 構造化結果

- 案件27通 → 構造化レコード**31件**（うち1通はアグリゲータ配信で人材12件＋案件5件が混在しており、案件5件のみ分解抽出。`row_key = messageId#1`〜`#5`）
- 空欄になりやすかった項目: `勤務地_県`（駅名のみで都道府県が明記されないケースが大半）、`必要経験年数`（複数条件併記で一意な数値化が困難）、`単価`（「応相談」「スキル見合い」のみで数値なし）
- いずれも「抽出不能は空欄・推測で埋めない」方針通りに処理

### 3.5 構造化コスト

- 80件処理でサブエージェント使用量 **約185,551トークン**（1通あたり約2,300トークン換算）
- ただしこれはエージェント実行のオーバーヘッド（ファイル読み書き等のツール呼び出し）を含んだ値であり、「1通の構造化に必要な最小コスト」そのものではない点に注意。既定モデル（Haiku採用）でのバッチ実行時の実コストとは別途測定が必要。

### 3.6 charset

- 明確な文字化け（mojibake等）は確認されず
- 軽微な異常として、1通の本文冒頭に配信システムのトラッキングIDと思われる孤立した数字行（「96」）が混入していたが実害なし

## 4. その他の観察事項

- **重複配信の実例**: FREE BRAIN社から同一人物（S.K・62歳・車載/PMOエンジニア）の人材情報がほぼ同一内容で同日中に2回配信（16:18・21:04）されているのを確認。CLAUDE.mdの「重複は多くない見込み」という前提は人材メールに限れば要再検証（案件メール側では今回のサンプルで重複は未確認）。
- **境界的な案件の扱い**: UIデザイナー・サウンドクリエイター募集など、厳密な「IT技術者」ではないが「求む」方向で準委任的な募集だったものは案件として構造化した（判断基準は職種でなく方向性）。
