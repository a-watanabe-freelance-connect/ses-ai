# SES-AI

ClaudeCode で **SES 営業を半自動化**するプロジェクト（SES＝IT技術者の常駐/準委任ビジネス。「案件」＝募集中の現場、「要員」＝稼働可能な技術者）。毎日大量に届くメールから**案件**を構造化してスプレッドシートに蓄積し（実現1）、**要員→案件**でマッチングする（実現2）。

> **本版は v0（最小スライス）が主役。** 堅牢化・最適化は末尾 **§backlog** に集約（詳細は `DECISIONS.md §6` と `TODO.md`）。まず v0 を端から端まで通す。
> 用語: 「将来」= MVP対象外（人材取込・方向B・`classify`・添付抽出）。「★」= マッチの核（案件/人材で対称。ただし**単価/経験は非対称**: 案件=上限予算 vs 人材=下限希望）。2026-07-01 の50体レビューは反映済み（要点は `DECISIONS.md`）。

## 目的（2つ）

1. **メール取込（MVPは案件）**: Gmail のメールを Claude が案件/人材/その他に仕分け、**案件を** `案件台帳` へ蓄積。
2. **マッチング（MVPは要員→案件）**: 自社要員の条件で `案件台帳` からマッチ案件を出し本人タブに蓄積。**許容条件プロンプト**（「経験3年だが優秀なので必要4年まで許容」「隣県も可」等）を都度加味。

## v0（今つくる最小スライス）

複数人運用あり・人員タブ記載あり。**カーソルは Gmail ラベルだけ**（時刻カーソルを使わない＝取りこぼし無し）で、まず端から端まで動かす。

- **取込**: Gmail を `-label:SES-AI/processed` で引き、**未処理メールを全件**取得（**fetch はフィルタしない**・案件/人材の判定は Claude）→ Claude が案件/人材/その他を推論で仕分け、**案件のみ**を主要列に構造化 → `案件台帳` に追記（**シートの `messageId` 列で重複回避**）→ 処理した各メールに `SES-AI/processed` ラベルを付与。**複数人で回してもラベルは共有メールボックス側にあり共有される**（担当をまたいで二重処理しない）。
- **マッチ**: `案件台帳` を**全件読む** → Claude が要員（`要員/*.md` の本人条件 or その場のプロンプト）にマッチ判定。許容条件（隣県可・経験±等）は**しきい値表でなく Claude が判断** → **ランキングを画面表示＋本人タブに 1行1案件で追記**（**本人タブ既存の `messageId` で重複回避**）。人員タブは事前用意（無ければ素朴に作成）。
- **スキーマ**: Claude 出力を `案件台帳` の主要列へ素直に流す。**`row_key` は `messageId` でよい**（分解や内容ハッシュは backlog）。
- **複数人**: 同時実行は稀。**`messageId` 重複回避（Gmail ラベル＋シートの messageId 列）で足りる**。ロック・カーソル・実行履歴の作り込みは不要（→ §backlog）。

## v0 実装契約（実装担当が最初に読む・迷わないための固定事項）

- **`messageId` = Gmail API の message `id`**（`users.messages.list/get` が返す不変ID。RFC822 Message-ID ではない）。取込・追記・本人タブ突合すべてこれで統一。
- **重複回避（v0・二段）**: ①主＝`SES-AI/processed` ラベル（付与済みは次回 `-label:processed` で取得されず＝再構造化されない）。②副＝案件台帳の `row_key` 列（append 前に存在チェック）。**1メールに複数案件のときだけ** `row_key = messageId#連番`（通常は `row_key = messageId`）。**副チェックは `row_key` 完全一致で判定**（`append_sheet.py` の実装。`messageId#1` と `messageId#2` は別 row_key として両方残る＝複数案件を取りこぼさない。前方一致にすると2件目以降が誤って重複扱いされるため不可）。※ 同時実行時の稀な二重行は v0 許容。なお `/ses-structure` の**突合確認**（triage の案件が parsed に構造化されたかの照合）は別処理で、そちらは messageId の**前方一致**（`messageId#連番` を拾う）で正しい。
- **構造化は v0 では"手動手順"**（スクリプト化しない）: **Claude Code セッションで Claude が `data/inbox` を読み `data/parsed` を書く**。Claude Code 経由なので API キー不要、コスト/キルスイッチは backlog。無人化（`claude -p`/API 化）は backlog。
- **fetch は v0 では本文テキスト抽出＋軽量クリーニング**（MIME デコード＋`text/plain` 優先、無ければ HTML→テキスト。`\r\n`正規化・署名/配信停止フッターの切除・追跡URLの`[link:ドメイン]`短縮・装飾セパレータ/連続空行の圧縮を実施。2026-07-06実測で本文平均14%削減、フッター重量メールは最大約79%削減）。可逆性は `案件メールリンク`（messageId から Gmail 原文へ復元可能）で担保。HTML表 TSV 化・Fwd/引用保持は引き続き **§backlog**。
- **中間データ契約（JSONL・1行=1レコード）**:
  - `data/inbox/*.jsonl` … `{messageId, from, subject, received_at(JST), body}`
  - `data/triage/*.jsonl` … `{messageId, 分類(案件|人材|その他), 理由}`（仕分け結果。ファイル名は対応する inbox と同名・全 messageId を網羅）
  - `data/parsed/*.jsonl` … `{row_key, <案件台帳の列名>: 値, …}`（キーは §スキーマの日本語見出しをそのまま。抽出不能は空文字）
  - `data/matches/<要員名>.jsonl` … マッチ結果（1行1案件。`write_match` が本人タブへ追記する入力）
- **SHEET_ID（既定）**: OAuth 完了後に **Claude が Sheets API で台帳を新規作成**し `.env` の `SHEET_ID` に記入（既存シートに合わせる場合のみ人間が ID を渡す）。
- **OAuth（v0）**: クライアント種別＝**Desktop app**、同意画面 User Type＝**External＋運用者をテストユーザーに追加**（Internal は backlog）。未審査アプリ警告は自分のアプリなので「詳細→安全でないページへ移動」で通す。`secrets/` は事前に作成。※ External＋Testing は refresh token が**約7日で失効**しうる→切れたら再同意（`token.json` 再生成）。頻繁なら早めに Internal 化（backlog）。
- **構造化のバッチ（v0）**: 手動で **1回あたり 50〜100通ずつ** `data/inbox` を Claude に読ませ `data/parsed` を書く（実測案件約1,850通/日なら20〜37バッチ相当に分割。"全件"＝全部を1回で、ではない）。1回の枚数はマイルストン0のコスト実測で確定。**実測規模（1日約5,455通・案件約1,850通/日）では手動バッチのみでの運用は非現実的**なため、無人化（既定Haiku＋Message Batches API）をv0完走後すぐの優先課題とする（§backlog）。
- **本人条件（要員プロフィール）は md ファイル** `要員/<名前>.md`（人間が用意・維持）。形式は簡単な箇条書き（`- スキル: Java, Spring, AWS` `- 経験年数: 5` `- 希望勤務地: 東京（隣県可）` `- 希望単価: 65` `- リモート希望: 一部可` `- 稼働開始: 2026-08`）。マッチ時に Claude がこれを読む。
- **人員タブ（スプレッドシート）はマッチ結果のみ**: 見出し行＋1行1案件を `write_match` が追記（タブ名＝要員名）。上部に本人条件は持たない（＝md 側）。
- **inbox↔parsed の突合**: `append_sheet` は `messageId` で inbox と parsed を join。`source_from`/`received_at` は inbox 由来、案件属性は parsed 由来を台帳の列へ書く。

## 規模の前提（設計を決める最重要ファクト）

- **受信量（2026-07-02実測・`MILESTONE0_REPORT.md`）: 1日 約5,455通（案件メールは概算約1,850通/日）/ 送信元 約1,500社**。当初想定「1日約1,000通」から約5.5倍の乖離が判明（レビュー2026-07-06 指摘 #3）。各社が独自配信するため**重複は多くない見込み**（案件メールでは実測で概ね裏付け。人材メールでは同一人物の重複配信を確認済み・§実現1対象外）。
- **MCP でのメール取込は不採用**（数千通規模の生本文を Claude のコンテキストに乗せるのは容量・コストとも非現実的）。→ 取得はスクリプト、構造化は Claude。**v0 は classify 無し＝fetch は全件取得し、Claude が全件から案件を拾う**（"絞り込み"＝classify は将来。v0 では手動バッチで捌く。→ §v0 実装契約）。
- **この Gmail は SES 専用受信箱＝届くのはほぼ案件/人材メール**（ノイズは極少だがゼロではない。実測でニュースレター等「その他」が数%混在）。よって **キーワード機械フィルタは作らず、案件/人材/その他判定は Claude の推論**。ただし**実体の無い純ノイズ配信元（ニュースレター等）は送信元ドメイン単位の軽量スキップリスト**（`fetch_gmail.py` の `SKIP_BODY_DOMAINS`）で本文全文読みを抑止する（メールは残しラベルは付与＝取りこぼしなし）。規模が上がったときの節約レバーは**汎用フィルタでなくバッチ化＋安いモデル（Haiku）＋キルスイッチ**（§backlog）。
- 実測結果（→ `MILESTONE0_REPORT.md`）: 案件/人材比率・案件1件あたり構造化コスト（約2,100〜2,300トークン/通、エージェント実行オーバーヘッド込み）・charset分布（明確な文字化けなし）。**手動50〜100通/バッチだと実測規模（案件約1,850通/日）で55〜110バッチ/日相当となり運用が破綻するため、無人化（Message Batches API＋Haiku）をv0完走後すぐの優先課題とする**（§backlog）。

## 運用ファイル構成

**複数人が別々のPCから実行する**。v0 の **進捗の正本＝Gmail ラベル `SES-AI/processed`＋シートの `messageId` 列**（共有メールボックス側にあるため自然に共有）。**ローカルに可変stateファイルを持たない**。

- **`TODO.md`** … 決めること・やること（可変トラッカー・現在地）。 **`DECISIONS.md`** … 確定した決定（レビュー要点も集約）。 **`CLAUDE.md`**（本ファイル）= 方針・設計・スキーマのみ。
- Claude は毎セッション最初に ① `TODO.md`（現在地）→ ② `DECISIONS.md` を読み戻す。

```

  構成図.html         # 構成図（ブラウザで開く。v0全体像）

  scripts/           # 薄いI/O【要実装】
    fetch_gmail.py   #   -label:processed で未処理取得 → data/inbox（可逆軽量化）
    append_sheet.py  #   messageId dedup で案件台帳へ追記＋ラベル付与
    read_sheet.py    #   マッチ用に案件/人員を読み出し（v0は全件）
    write_match.py   #   マッチ結果を人員タブへ追記（messageId dedup）
    verify_triage.py #   inbox↔triage の messageId 突合検証（OS非依存。append前の必須チェック）

  data/inbox|triage|parsed|matches/ # 生メール／仕分け結果／構造化結果／マッチ結果（各追記前。処理済みは _archive/<日付>/ へ退避可）

  secrets/           # Google OAuth の JSON（credentials.json / token.json。★git対象外）
  .claude/skills/    # /ses-triage・/ses-structure・/ses-match（3スキル構成＝DECISIONS §8。仕様は docs/skills/）
```

**台帳スプレッドシート**（クラウド側・共有の正本）のタブ構成（MVP）:
- `案件台帳` … 案件の本体（下記スキーマ・`messageId` 列を持つ）
- `人員一覧` … 自社要員の名簿（索引: 名前＋各人タブへのリンク＋件数）。1行1人。
- `<人員ごとのタブ>` … 1人1タブ（タブ名＝要員名）。方向Aのマッチ案件を1行1案件で蓄積。無ければ素朴に作成。本人条件は `要員/*.md`（→ §v0実装契約）。
- `_state` … 予算上限・キルスイッチ・セッション単位ロック（key-value。cron化フェーズ1で実装済み。`scripts/state.py`）。
- `_runlog` … fetch/append の監査ログ（2フェーズ・追記のみ。cron化フェーズ1で実装済み。`scripts/state.py`）。
- **【backlog/将来】**: `スキル正準辞書`/`_staging`/`_quarantine`/`_archive`（堅牢化）／`人材台帳`・`マッチ結果`・`_allowlist`・`_dropped_sample`（将来）。→ §backlog

## アーキテクチャ（運ぶ・絞るのはコード、読む・判断するのは Claude）

deterministic な処理（取得・重複回避・書込）はスクリプト（**Python**）、非構造テキストの**解析**と**マッチ判断**は Claude。数千通/日規模を丸ごと読ませないための段構え。

```
[Gmail 〜5,455通/日（実測。うち案件約1,850通/日）]
   │ (1) 未処理取得: fetch_gmail -label:SES-AI/processed（本文テキスト抽出・ページング/バックオフ）
   ▼
data/inbox/*.jsonl（本文テキスト抽出）
   │ (2) 構造化: Claude が inbox を読み案件/人材/その他を推論で仕分け → 案件のみ構造化（機械フィルタ無し）
   ▼
data/parsed/*.jsonl（案件・スキーマ準拠）
   │ (3) 追記＋dedup: append_sheet（シートの messageId 列で重複回避 → 追記 → ラベル付与）
   ▼
[スプレッドシート: 案件台帳]  ← 実現2(要員→案件)はここを全件読む
```

取得（大量）・書込・読取はスクリプトで確定。対話中の軽い確認だけ Sheets MCP 併用可。

## 連携方法と認証

第一選択は Google API を叩く自作スクリプト。OAuth ユーザートークン1本で Gmail・Sheets を賄う。

- 必要スコープ: `gmail.modify`（読取＋ラベル付与。ラベル付与に modify 必須）／`spreadsheets`。
- 認証情報は `secrets/credentials.json`（Google の JSON。`.env` にはできない＝ライブラリが JSON を読む）→ 初回同意で `secrets/token.json`。
- 設定値・素の鍵は **`.env`**（`SHEET_ID`・対象Gmail・creds のパス／無人化時のみ `ANTHROPIC_API_KEY`）。`.env`・`secrets/` とも gitignore、テンプレは `.env.example`。
- フォールバック/補助: Sheets MCP は対話時の軽い読取のみ。
- **堅牢化（OAuth User Type=Internal・委任アクセス・PII 保持/マスク/削除）は §backlog。**

## 実現1: 案件メール取込

### フロー（v0・4段）
1. `fetch_gmail.py` で **`-label:SES-AI/processed` の未処理だけ**取得 → `data/inbox`（**本文テキスト抽出**＝MIMEデコード＋`text/plain` 優先。ページング・指数バックオフ）。**カーソルは Gmail ラベルだけ**（前回の時刻を持たない＝取りこぼし無し）。ただし `_state.daily_budget`（既定8000件/日・セーフティネット）を超える要求は超過分が部分取得になる（cron化フェーズ1）。※ 可逆軽量化・HTML表TSV化・Fwd保持は §backlog。
2. **機械フィルタ（classify）は作らない**（SES 専用受信箱＝ノイズ除去不要。判定は次の 3 の Claude 推論）。
3. Claude が inbox を読み **案件/人材/その他を推論で仕分け** → **案件だけスキーマに構造化**（`data/parsed/`）。人材はスキップ、その他は破棄しログ。1通に複数案件は分解して1行ずつ。※ 案件と人材は語彙が同じで違いは方向（求む＝案件／提案＝人材）。**Claude は意味判断のみ・キー生成しない**。
4. `append_sheet.py` が **シートの `messageId` 列で重複回避して `案件台帳` に追記** → 該当メールに `SES-AI/processed` ラベルを付与（この順）→ **`fetch_gmail.py`が取得したセッション単位ロックをここで解放**（cron化フェーズ1。§v0実装契約）。

> 差分カーソルの overlap 化・row_key 内容ハッシュ・upsert・コミット順序固定・ステータス更新反映（四分類）・NFKC 正規化・添付スキルシート抽出 は **§backlog / 将来**。`_state`/`_runlog`（予算上限・キルスイッチ・ロック・監査ログ）はcron化フェーズ1で実装済み（DECISIONS §6-1）。

## 実現2: マッチング（要員 → 案件）※MVPは方向Aのみ

v0 は `案件台帳` を全件読み、許容条件（隣県可・経験±等）は **Claude が判断**する。人員タブは事前用意（無ければ素朴に作成）。

### 方向A: 要員 → 案件
- 入力: `要員/<名前>.md` の本人条件＋許容条件（その場のプロンプトで上書きも可）。
- 処理: `案件台帳` を**全件読む** → Claude 評価 → ランキング。
- 記録: マッチ案件を本人タブへ **1行1案件で追記**（列＝ 記載日 / 適合度 / 案件名 / 単価 / 勤務地 / リモート / 商流 / 必須スキル / 適合理由 / 懸念 / ステータス / 案件row_key(messageId) / リンク）。**本人タブ既存の `messageId` で重複回避**、再実行は新規のみ・手動ステータス保持。適用した許容条件も結果行に記録（再現性）。

### 評価軸（★対称・単価/経験は非対称）
必須スキル充足 / 経験年数（許容込み）/ 勤務地・居住地（隣県許容込み）/ 単価（**案件=上限予算 vs 人材=下限希望**。区間の重なりで判定）/ リモート / 開始時期 / 商流。
- **国籍・年齢・性別は機械足切りにしない**（職業安定法・労働施策総合推進法9条）。参考情報＋適法性要確認の警告を出力に添える。

### 出力
1件1ブロック: 適合度（◎/○/△）＋案件名＋単価/勤務地/リモート/商流＋適合理由（許容条件の適用を明記）＋懸念・要確認＋元メールリンク。必要なら「惜しくも外した候補」も別枠。

> 粗フィルタ緩和（隣接県表・空欄通し）・鮮度TTL・スキル正準辞書・`person_id`/行ロック・単一マッチ結果タブへの移行（人員タブ>30）・**方向B（案件→人材）** は **§backlog / 将来**。
