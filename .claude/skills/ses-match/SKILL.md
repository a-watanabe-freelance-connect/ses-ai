# /ses-match — 要員→案件マッチング

CLAUDE.md §実現2 に対応するSkill。v0は方向A（要員→案件）のみ。しきい値表による機械的フィルタではなく、**Claudeが都度判断**する。

## 前提

- 対象要員の `要員/<名前>.md` が用意されていること（未整備なら本人へ確認、または既存スキルシート等の資料があればテンプレート `要員/_テンプレート.md` を読み込んで埋める）
- `.env` に `SHEET_ID` 設定済み、OAuth 設定済み（`secrets/credentials.json`・`secrets/token.json`）、`.venv` 構築済み

## 手順

### 1. 案件台帳を全件読む

```
.venv/bin/python scripts/read_sheet.py       # macOS/Linux
.venv\Scripts\python.exe scripts\read_sheet.py  # Windows
```

- ヘッダー行をキーにしたdictのリストをJSON出力する（列構成の変更に自動追従）
- 出力はシェルの`>`リダイレクトではなく、Pythonスクリプト側で`encoding="utf-8"`のファイル書き込みを行い、Readツールで読むこと（文字化け防止）
- 件数が多い場合、直接読んで評価するかサブエージェントに委任するかは件数次第で判断してよいが、マッチング判断は業務理解が要るため基本は自分（メインの会話）で評価する方が精度が高い

### 2. 要員プロフィールの確認

`要員/<名前>.md` を読む。加えて、その場で許容条件のプロンプト（例:「経験3年だが優秀なので4年要件まで可」「隣県も可」「単価60万以上なら可」）が与えられていれば、本人条件に対する上書き・追加条件として適用する。

### 3. 評価

**評価軸**（★=対称・単価と経験は非対称）:
- 必須スキル充足
- 経験年数（許容込み）
- 勤務地・居住地（隣県許容込み）
- 単価（**案件=上限予算 vs 人材=下限希望**。区間の重なりで判定）
- リモート
- 開始時期
- 商流

**判定ルール**:
- 国籍・年齢・性別は**機械的な足切りにしない**（職業安定法・労働施策総合推進法9条）。案件側に制限記載があっても、参考情報として懸念欄に記すに留め、適法性要確認の警告を添える
- 適合度は ◎（強い一致）／○（好条件だが一部懸念）／△（部分一致・要確認点が多い）の3段階
- 明確に不一致（言語・職種・経験年数が大きく乖離等）な案件は対象外としてよい。ただし「惜しい」候補（スキルは合うが勤務地がネック等）は積極的に△として残す（機械的除外よりは人間の判断に委ねる）
- 適合理由・懸念は、要員のプロフィール（経歴・希望条件）の具体的な根拠と紐付けて記述する（例:「PJ4の出荷管理システム経験がこの物流案件のドメインと一致」）
- **適用した許容条件は適合理由に明記**する（再現性のため。結果行に残る）

### 4. マッチ結果の書き出し

`data/matches/<名前>.jsonl` に1行1案件で追記する。既存ファイルがあれば末尾に追記（`write_match.py`側で`row_key`による重複排除が行われるため、同じ案件を再掲しても害はない）。

各レコードのフィールド:
```
row_key（案件台帳のrow_key＝messageId）, 適合度（◎/○/△）, 案件名, 単価, 勤務地,
リモート, 商流, 必須スキル, 適合理由, 懸念, リンク（案件メールリンク）
```

### 5. 本人タブへ反映

```
.venv/bin/python scripts/write_match.py <要員名>       # macOS/Linux
.venv\Scripts\python.exe scripts\write_match.py <要員名>  # Windows
```

- 本人タブ（タブ名＝要員名）の列: `記載日 / 適合度 / 案件名 / 単価 / 勤務地 / リモート / 商流 / 必須スキル / 適合理由 / 懸念 / ステータス / 案件row_key(messageId) / リンク`
- `記載日`＝実行日、`ステータス`＝未指定なら「未対応」をスクリプトが自動設定する
- タブが存在しなければ見出し行付きで自動作成する
- **既存の`案件row_key(messageId)`列で重複回避**（再実行は新規のみ・既存行の手動ステータスは保持）
- **注意**: 手動でタブを事前作成していた場合、見出し行が無いとデータが1行目から書かれてしまうバグが過去に発生した（`_ensure_tab`で見出し有無を確認するよう修正済みだが、念のため実行後にタブの1行目が見出しになっているか確認する）

## 完了後の報告（画面表示ルール・2026-07-03決定）

冒頭で必ず ◎/○/△ それぞれの件数サマリーを示す。詳細ブロックの表示範囲は **◎の件数**で切り替える（コンテキスト消費とのバランスのため）。

- **◎が10件以内**: ◎と○を1件1ブロックでフル表示。△は件数のみ＋「詳細は本人タブでご確認ください」
- **◎が10件超**: ◎のみ1件1ブロックでフル表示。○・△は件数のみ

各ブロックのフォーマット（CLAUDE.md準拠）:
```
### <適合度> <案件名>
- 単価: XX万円 / 勤務地: XXX / リモート: XXX / 商流: XXX
- 必須スキル: XXX
- 適合理由: XXX（許容条件を適用した場合は明記）
- 懸念・要確認: XXX
- リンク: <案件メールリンク>
```

## エラー・例外

- `要員/<名前>.md` が無い → マッチングを実行せず、本人確認またはテンプレート（`要員/_テンプレート.md`）からの作成を案内。
- マッチ結果ファイル無しで `write_match.py` を実行 → FileNotFoundError（先に評価・書き出しを行う）。
- 案件台帳が空 → 「先に `/ses-triage` → `/ses-structure` で取込を」と案内。
