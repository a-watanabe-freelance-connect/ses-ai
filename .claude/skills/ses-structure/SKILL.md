# /ses-structure — 案件メールの構造化

CLAUDE.md §実現1・§スキーマ・§v0実装契約 に対応するSkill。`/ses-triage` で「案件」と仕分けられたメールを `案件台帳` のスキーマへ構造化し、`append_sheet.py` で台帳へ追記・`SES-AI/processed` ラベルを付与する。v0パイプラインの第2段（**実現1がここで完結**）。正本仕様は `docs/skills/ses-structure.md`（本SKILLは仕様書に準拠。仕様変更は仕様書を先に直す）。

```
data/inbox + data/triage(分類=案件) → 【/ses-structure 本スキル】 → data/parsed
                                        → append_sheet.py（台帳追記 dedup → inbox全件へラベル付与）
```

## 重要な不変条件（先に読む）

- **ラベル付与は最終段 `append_sheet.py` が inbox 全件（人材/その他含む）へ一括で行う**。案件だけでなく inbox の全メールに `SES-AI/processed` が付く（案件でないメールが毎回再取得・再判定され続けるのを防ぐため）。
- **`append_sheet.py` は `data/inbox`・`data/parsed` をディレクトリ全glob する**（バッチ指定不可）。対象は「今回分」でなく `data/inbox` の現存全ファイル。前バッチのファイルは append 前に `data/_archive/` へ退避しておくのが安全（手順4・7）。
- **append 前の突合確認は必須**（手順4）。未仕分け・未構造化のメールが残ったまま append_sheet を実行すると、そのメールにもラベルが付いて次回以降取得されなくなる（＝取りこぼし）。未了があれば append_sheet を**実行せず報告して停止**する。
- **抽出不能は空文字**（推測で埋めない）。

## 前提

- `/ses-triage` 完了（`data/triage/` が `data/inbox/` の全ファイル・全 messageId をカバー）
- `.env` に `SHEET_ID` 設定済み、OAuth 設定済み（`secrets/credentials.json`・`secrets/token.json`）、`.venv` 構築済み

## 入力（データ契約）

- `data/inbox/*.jsonl` … `{messageId, from, subject, received_at, body}`（本文の供給元）
- `data/triage/*.jsonl` … `{messageId, 分類, 理由}` → **`分類=案件` の messageId のみ**が構造化対象。人材・その他はスキップ（ラベル付与だけ後段で行われる）。

## 手順

### 1. 対象抽出

`data/triage/*.jsonl` から `分類=案件` の messageId 一覧を作る。人材・その他はスキップ。

### 2. 構造化（50〜100通/バッチ）

対象 messageId に対応する inbox レコードの `body` を 50〜100通/バッチ で読み、§構造化規約 に沿って `案件台帳` の列へ構造化する。1通に複数案件があれば分解して1行ずつ（row_key の採番は §row_key に従う）。

### 3. 書き出し

`data/parsed/<inboxと同じファイル名>.jsonl` へ1行1案件で出力（Write ツール＝UTF-8。文字化け防止）。キーは `案件台帳` の日本語見出しをそのまま使う。

### 4. 突合確認（append 前の必須チェック）

**`append_sheet.py` は `data/inbox/*.jsonl`・`data/parsed/*.jsonl` を _ディレクトリ全glob_ する**（引数でバッチ指定できない・v0）。よって突合の対象は「今回のバッチ」ではなく **`data/inbox` に現存する全ファイルの全 messageId**。前バッチの処理済みファイルが残っていると、それらも再走査・再ラベルの対象になる。

`append_sheet.py` を実行する**前に**、以下を確認する:

- inbox の全 messageId が triage 済みか（`scripts/verify_triage.py` で inbox↔triage を機械照合。未仕分けの inbox ファイル／messageId が無いか）。
- triage `分類=案件` の全 messageId が parsed に存在するか（row_key の messageId 前方一致で照合）。

**未了があれば append_sheet.py を実行せず、不足分（未仕分けファイル／未構造化 messageId）を報告して停止する。** 理由: append_sheet はラベルを inbox **全件**に付与するため、未仕分け・未構造化のメールが残った状態で実行すると、そのメールは次回以降取得されなくなる（＝取りこぼし）。

**推奨: バッチ完了ごとに処理済みの inbox/triage/parsed を `data/_archive/<YYYYMMDD>/` へ退避してから append_sheet.py を実行する**（手順7を先に行う）。`data/inbox` を「今回処理する分だけ」に保つのが、全glob由来の混入・取りこぼしを防ぐ最も確実な方法。

### 5. 追記＋ラベル（append_sheet.py）

突合確認をクリアしたら:

```
.venv/bin/python scripts/append_sheet.py       # macOS/Linux
.venv\Scripts\python.exe scripts\append_sheet.py  # Windows
```

- `案件台帳` へ `row_key` 列の重複チェック付きで追記（既存 row_key はスキップ）。
- `ingested_at`／`received_at`／`source_from`／`source_email`／`案件メールリンク`（未指定時）はスクリプトが inbox から自動補完する。
- 追記後、inbox 全メール（人材/その他含む）へ `SES-AI/processed` ラベルを一括付与。
- **cron化フェーズ1以降、正常完了時に `fetch_gmail.py` が取得したセッションロックをここで解放する**（1バッチサイクルの終了点）。
- 出力例: `Appended N rows to 案件台帳.` / `Labeled L messages as SES-AI/processed.`

### 6. 報告

§完了報告 の形式で画面表示。

### 7. 後始末（退避。次バッチの取りこぼし防止のため実質必須）

処理済みの `data/inbox` / `data/triage` / `data/parsed` を `data/_archive/<YYYYMMDD>/` へ退避する。台帳の dedup により二重行は生じないが、退避しないと次回 append 時に**古いファイルも全glob対象**として再走査・再ラベルされ、突合確認（手順4）の負荷と取りこぼしリスクが積み上がる。理想は「append 直前に手順7を行い、`data/inbox` を今回分だけに保つ」運用。**将来はバッチ/対象ファイル指定引数を append_sheet.py に持たせ、全glob依存を下げる（backlog）。**

## 構造化規約

### row_key（一次キー）

- 通常: `row_key = messageId`。
- 1メールに複数案件: 本文の出現順に `messageId#1`, `messageId#2`, … と**決定的に**採番（再実行しても同じ番号になること。append_sheet の重複判定は row_key の一致で行うため、採番が揺れると二重行になる）。
- row_key は messageId から機械的に導出する。内容ハッシュ等の独自キーを発明しない（内容ハッシュ化は backlog）。

### 共通原則

- **抽出不能は空文字**（推測で埋めない）。NFKC 正規化・スキル正準辞書・値バリデーションは backlog。
- `ingested_at`／`received_at`／`source_from`／`source_email` は parsed に**書かない**（append_sheet が inbox 由来で上書き補完する）。

### 列別の抽出規約

| 列 | 規約 |
|---|---|
| `案件名` | 件名または本文の案件見出しを簡潔に（複数案件分解時は各案件の見出し） |
| `商流` | 元請/1次/2次…（「エンド直」「元請直下」等の記載から。不明は空欄） |
| `必須スキル` / `尚可スキル` | 原文の表記を保った列挙（読点・カンマ区切り。正準化は backlog） |
| `必要経験年数` | 数値（年）。「3年以上」→ `3`。スキル別の年数要件は必須スキル内に付記 |
| `単価` | 万円/月の**上限予算**を数値で。「〜80万」「MAX80」→ `80`、「70〜80万」→ `80`（min/max 分離は backlog。下限は生文抜粋/備考で追える） |
| `精算幅` | 原文表記（例 `140-180h`、`150h～200h`） |
| `勤務地_県` | 本文に都道府県名が明記されていればそのまま。**駅名・市区名からの県補完は行わない**（`append_sheet.py` が機械的に補完する。§正規化層を参照）。明記が無ければ空欄でよい |
| `勤務地_詳細` | 市区・最寄駅（原文表記） |
| `リモート` | 原文の表記をそのまま（例「フル」「一部（週1出社）」「基本常駐」）。**enum化（フルリモート/一部リモート/常駐）は行わない**（`append_sheet.py` が正規化する） |
| `開始時期` | 原文相当の表記でよい（例「8月」「2026年8月」「即日」）。既に `YYYY-MM` が明確な場合はそれでもよい。**年欠落の補完・複数月併記からの最早月選定は行わない**（`append_sheet.py` が正規化する） |
| `期間` / `面談回数` / `募集人数` | 原文の値 |
| `国籍_年齢制限` | 記載があれば内容をそのまま。**参考情報**（マッチで機械足切りに使わない） |
| `契約形態` | 準委任/派遣 等 |
| `ステータス` | 新規は「募集中」 |
| `案件メールリンク` | 空欄可（append_sheet が Gmail リンクを自動補完） |
| `生文抜粋` | 検証用に条件記載部分の原文を短く（〜200字目安） |
| `備考` | 上記に収まらない特記（個人事業主可否・所属制限・支援費等） |

### 正規化層（`append_sheet.py` が自動で行う。Claude は関与しない）

同一の原文でもモデルの解釈揺れで正規化結果がバッチ間でばらつく問題（レビュー2026-07-06）に対応するため、以下は `append_sheet.py` 側の決定的変換に一本化した。Claude は原文相当の抽出だけを行い、下記の正規化は意識しなくてよい。

- **開始時期**: `YYYY-MM` または「即日」へ変換。年が無い場合は `received_at` の年で補完。複数月併記（例「5月/6月/7月」）は最早月を採用。パターン未対応の原文はそのまま残る。
- **勤務地_県**: 空欄のときだけ `勤務地_詳細` の駅名・市区名から対応表で機械補完（`scripts/append_sheet.py` の `AREA_TO_PREFECTURE`）。未収録の地名は空欄のまま（推測しない）。`勤務地_詳細` が非空なのに `勤務地_県` が確定しない場合は `append_sheet.py` 実行時に `[warn]` 表示される（対応表への追記要否の判断材料）。
- **リモート**: `{フルリモート/一部リモート/常駐}` のいずれかを先頭に付け、原文の限定条件は括弧書きで保持（例「一部リモート（週2回程度出社）」）。

## エラー・例外

- `Label not found: SES-AI/processed` → `scripts/setup_v0a.py` を先に実行（Windows: `.venv\Scripts\python.exe scripts\setup_v0a.py` / macOS・Linux: `.venv/bin/python scripts/setup_v0a.py`）。
- `SHEET_ID` 未設定（KeyError）→ `.env` を確認。
- 同時実行による稀な二重行は v0 許容（DECISIONS §4。upsert・行ロックは backlog）。
- 突合確認（手順4）で未了検出 → append_sheet を実行せず、不足分（未仕分けファイル／未構造化 messageId）を報告して停止。
- **`kill_switch=on のため中断しました`** → 人間が意図的に全処理を止めている状態。追記・ラベル付与とも行われない。`python scripts/state.py killswitch off` で解除してから再実行する。
- **append_sheet.py 完了後もロックが解放されない（`[warn] ロックは自分の保持ではなかった`）** → 別の owner が既にロックを握っている（別バッチが並行している等）。`python scripts/state.py lock status` で確認する。
