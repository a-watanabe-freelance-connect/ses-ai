# SES-AI 運用マニュアル（操作手引き）

このドキュメントは、**SES-AI を毎日回す運用者**のための操作手引きです。「メールを取り込んで案件を台帳に貯める」→「要員に合う案件を出す」までを、**迷わず実行できる**ことを目的にしています。

> このマニュアルは**日々の操作**にフォーカスしています。新しいPC・新しい担当者の**初期セットアップ**（Google Cloud / OAuth / `.env` / venv）は範囲外です。設計思想や決定の背景は [`CLAUDE.md`](../CLAUDE.md) / [`DECISIONS.md`](../DECISIONS.md) を参照してください。

---

## 0. これは何をするシステムか

毎日大量に届く SES メールを、**Claude（Claude Code）と3つのスキルで半自動処理**します。

| やること | スキル | ひとことで |
|---|---|---|
| **実現1: 案件取込** | `/ses-triage` → `/ses-structure` | 未処理メールを取得 → 案件/人材/その他に仕分け → 案件を「案件台帳」に構造化して追記 |
| **実現2: マッチング** | `/ses-match` | 案件台帳を全件読んで、要員に合う案件を本人タブに蓄積 |

**大事な前提（この2つを覚えておけば事故らない）**

1. **操作は「Claude に日本語で話しかける」だけ。** スクリプトを直接叩く必要はありません。スキルが `fetch_gmail.py` などを裏で実行します。運用者の役目は、**正しいタイミングでスキルを起動し、確認ポイントをチェックすること**です。
2. **進捗は全員で共有される。** 処理済みの目印は Gmail のラベル `SES-AI/processed` と、シートの `messageId` 列にあります。どちらも共有メールボックス／共有シート側にあるので、**複数人が別々のPCから回しても二重処理になりません**。

## 1. 毎日の全体像（1枚で）

```
 ┌─ 実現1: 案件取込 ────────────────────────────────────────┐
 │                                                          │
 │  ①「新着を取り込んで仕分けして」                          │
 │      → /ses-triage                                       │
 │        fetch_gmail.py（未処理メール取得）→ data/inbox    │
 │        Claude が 案件/人材/その他 に仕分け → data/triage  │
 │        verify_triage.py（inbox↔triage 突合）             │
 │                                                          │
 │  ②「仕分け済みの案件を構造化して台帳に追記して」          │
 │      → /ses-structure                                    │
 │        Claude が案件を構造化 → data/parsed               │
 │        突合確認（未処理が残っていないか）                 │
 │        append_sheet.py（台帳追記 + 全メールにラベル付与） │
 │                                                          │
 └──────────────────────────────────────────────────────────┘
 ┌─ 実現2: マッチング ──────────────────────────────────────┐
 │                                                          │
 │  ③「この要員に合う案件を出して」/「全員マッチングして」   │
 │      → /ses-match                                        │
 │        read_sheet.py（案件台帳を全件読む）               │
 │        read_profile.py（要員プロフィール取得）           │
 │        Claude が評価・ランキング                          │
 │        write_match.py（本人タブに追記）                  │
 │                                                          │
 └──────────────────────────────────────────────────────────┘
```

> **① と ② は必ず続けて実行してください。** ① を始めると、そのバッチが終わるまで**セッションロック**がかかり、他の人の取込がブロックされます。ロックは ② の `append_sheet.py` 完了で自動的に解放されます。「仕分けだけやって放置」は避けてください（→ [困ったとき: ロック](#ロックが取れない付与できない)）。

## 2. 作業① 案件を取り込んで仕分ける（/ses-triage）

### 起動のしかた

Claude Code のセッションで、こう話しかけます（どれでもOK）:

- 「新着メールを取り込んで仕分けして」
- 「inbox を仕分けして」
- 「今日の新着を取り込んで」

すると `/ses-triage` が起動し、次を順に行います:

1. **取得**: `fetch_gmail.py` が `SES-AI/processed` ラベルの**付いていない**メールを取得し `data/inbox/` に保存します。
2. **仕分け**: Claude が本文を読んで、各メールを **案件 / 人材 / その他** に分類し `data/triage/` に書き出します（キーワードの機械フィルタではなく Claude の推論）。
3. **突合検証**: `verify_triage.py` で「inbox の全メールが漏れなく仕分けされたか」を機械チェックします。

### 件数を絞りたいとき（テスト・実測）

プロンプトで指定できます。スキルが対応する引数を付けて実行します。

| 言い方の例 | 効果 |
|---|---|
| 「まず50通だけ取り込んで仕分けして」 | `--limit 50`（取得上限） |
| 「7月3日のメールだけ取り込んで」 | `--date 2026-07-03`（単日のみ） |
| 「7月1日以降だけ対象にして」 | `--since 2026-07-01`（この日以降。過去の全履歴を巻き込まないフロア） |

### ここを確認（完了報告のチェックポイント）

`/ses-triage` は最後に件数サマリを出します。次を確認してください:

- **合計 X 通 ＝ 案件 A / 人材 B / その他 C** の内訳が妥当か（例: 案件が極端に少ない → 仕分けミスや取得漏れの疑い）。
- 「その他」の内訳例に、**本来は案件/人材のはずのメールが混ざっていないか**。
- **`verify_triage.py` が NG を出していないか**（NG のときは該当ファイルを作り直します。Claude が案内します）。

> **重要**: この段階では**まだラベルは付きません**。ラベル付与は次の ② の最終段でまとめて行われます。だから ① で止めてもメールは「未処理」のまま残り、取りこぼしは起きません。ただしロックは掛かったままなので、**続けて ② に進んでください**。

## 3. 作業② 案件を構造化して台帳に追記する（/ses-structure）

### 起動のしかた

① に続けて、こう話しかけます:

- 「仕分け済みの案件を構造化して台帳に追記して」
- 「今日の案件を台帳に入れて」

`/ses-structure` が起動し、次を順に行います:

1. **構造化**: `分類=案件` のメールだけを、案件台帳のスキーマ（案件名・必須スキル・単価・勤務地…全21列）に構造化し `data/parsed/` に書き出します。1通に複数案件があれば分解して1行ずつになります。
2. **突合確認（必須の安全弁）**: append する前に、「未仕分け・未構造化のメールが残っていないか」を確認します。残っていれば**追記せず停止**します。
3. **追記＋ラベル**: `append_sheet.py` が案件台帳に重複回避しながら追記し、**inbox の全メール（人材・その他も含む）に `SES-AI/processed` ラベルを付与**します。ここでセッションロックも解放されます。

### なぜ人材・その他にもラベルが付くのか

案件でないメールにラベルを付けないと、**毎回それらを取得・再判定し続けてしまう**からです。全メールに「見た」印を付けることで、次回の取得対象から外れます（＝案件でないメールも「処理済み」として前に進む）。案件台帳には案件しか入りません。

### ここを確認（完了報告のチェックポイント）

- **追記 N 行 / 重複スキップ M 行 / ラベル付与 L 通** の数字が想定どおりか。
  - `ラベル付与 L` が **inbox の総数と一致**しているか（多すぎ・少なすぎは古いファイルの混入疑い → [困ったとき](#4-困ったときトラブル対処)）。
- 抽出できず空欄が目立った列の言及があれば、スキーマ改善の参考にします（対応は任意）。
- `[warn]` が出ていないか（列の欠落・勤務地の県が補完できなかった等。多くは無害ですが、頻発するようなら記録を）。

### ⚠️ append_sheet 実行前の鉄則

`append_sheet.py` は `data/inbox` / `data/parsed` を**フォルダごと全部**読みます（「今回分だけ」の指定ができません）。そのため:

- **前回バッチの古いファイルが `data/inbox` に残っていると、それも巻き込んで再ラベルされます。**
- スキルは通常、処理済みファイルを `data/_archive/<日付>/` に退避してから append します。**「data/inbox に今回処理する分だけがある」状態を保つ**のが取りこぼし防止の要です。
- 過去に、退避し忘れた別バッチ100通が一緒にラベル付与された事例があります。**append 直前に `data/inbox` の中身を確認する習慣**をつけてください（不安なら Claude に「append の前に data/inbox に何が入っているか確認して」と依頼）。

## 4. 作業③ 要員に合う案件を出す（/ses-match）

### 2つのモード

| モード | 起動の言い方 | 動き |
|---|---|---|
| **対象指定** | 「渡辺さんに合う案件を出して」 | 指定した1名だけをマッチング。その場で許容条件も指定可 |
| **一括** | 「稼働中の全員をマッチングして」 | 「稼働中」フォルダの全要員を1名ずつ順に処理 |

### 許容条件をその場で足せる（対象指定モードのみ）

マッチングは、しきい値表ではなく **Claude が都度判断**します。だから自然文で条件を上乗せできます:

- 「経験3年だが優秀なので**4年要件まで可**、東京＋隣県で」
- 「単価60万以上なら可」
- 「リモート必須の案件だけ」

適用した許容条件は、結果行の「適合理由」に明記されます（あとで再現・確認できるように）。

### 何が起きるか

1. `read_sheet.py` が案件台帳を**全件**読みます。
2. `read_profile.py` が要員プロフィール（Google Drive「稼働中」フォルダ内の `<要員名>.md`）を取得します。
3. Claude が評価軸（必須スキル / 経験年数 / 勤務地・隣県 / 単価 / リモート / 開始時期 / 商流）で採点し、**◎ / ○ / △** でランク付けします。
4. `write_match.py` が結果を**本人タブ**（タブ名＝要員名）に1行1案件で追記します（既存の案件は重複回避）。

### ここを確認

- 画面に **◎ / ○ / △ の件数サマリ**が出ます。◎が10件以内なら◎○を詳細表示、10件超なら◎のみ詳細表示（残りは本人タブで確認）。
- **国籍・年齢・性別は機械的な足切りに使いません**（職業安定法・労働施策総合推進法9条）。案件側に制限記載があっても参考情報として懸念欄に記し、適法性要確認の警告が添えられます。これは仕様です。
- 本人タブの**1行目が見出し行**になっているか（手動でタブを作っていた場合、見出しが無いとデータが1行目から書かれる不具合が過去にありました）。

## 5. 困ったとき（トラブル対処）

代表的なメッセージと対処です。多くは Claude が対処法を案内しますが、判断の参考にしてください。

### ロックが取れない／付与できない

- **`[abort] ロック取得に失敗しました`**
  - 誰かが取込（`/ses-triage`→`/ses-structure`）を**途中まで実行して保持中**です。相手が完了するのを待つのが基本。
  - 明らかに放置（誰も作業していないのにロックが残っている）なら、`_state` タブの `lock_owner` / `lock_acquired_at` を確認のうえ、強制解除できます:
    ```
    .venv\Scripts\python.exe scripts\state.py lock status          # 状態確認
    .venv\Scripts\python.exe scripts\state.py lock release --force # 強制解除
    ```
  - ロックは既定6時間（TTL）で自然解放もされます。

### 本日の上限に達した

- **`[abort] 本日の daily_budget を使い切っている`**
  - 1日の取得上限（既定8000件・セーフティネット）に達しました。
    ```
    .venv\Scripts\python.exe scripts\state.py budget status   # day_spent / daily_budget を確認
    ```
  - 異常な消費でなければ翌日まで待つか、必要なら `_state` タブの `daily_budget` を人が調整します。

### 全処理が止まっている

- **`kill_switch=on のため中断しました`**
  - 誰かが意図的に全処理を止めています（緊急停止）。理由を確認してから解除:
    ```
    .venv\Scripts\python.exe scripts\state.py killswitch status  # 状態確認
    .venv\Scripts\python.exe scripts\state.py killswitch off     # 解除
    ```

### 認証エラー（トークン切れ）

- **`fetch_gmail.py` が認証エラーで失敗する**
  - 現在の認証（OAuth External+Testing）は refresh token が**約7日で失効**することがあります。再同意でトークンを作り直します:
    ```
    del secrets\token.json
    .venv\Scripts\python.exe scripts\auth.py    # ブラウザで sales@freelance-connect.jp にログインして同意
    ```
  - 未審査アプリ警告が出たら「詳細 → 安全でないページへ移動」で進みます（自分たちのアプリなので問題ありません）。

### ラベルが見つからない

- **`Label not found: SES-AI/processed`**
  - Gmail 側にラベルがまだ作られていません。一度だけ作成スクリプトを実行:
    ```
    .venv\Scripts\python.exe scripts\setup_v0a.py
    ```

### 「未処理メールなし」と出た

- **`No unprocessed messages.`** は正常です。取り込むべき新着が無いだけ。そのまま終了して構いません。

### 台帳が空でマッチングできない

- 先に ①→② で案件を取り込んでください（`/ses-triage` → `/ses-structure`）。

### ラベル付与数が inbox 総数と食い違う／古いファイルが混ざった疑い

- `data/inbox` に前回バッチの残りが混ざっている可能性があります。Claude に「data/inbox の中身を確認して、今回処理していない古いファイルがあれば `data/_archive` に退避して」と依頼してください（→ [作業②の鉄則](#-append_sheet-実行前の鉄則)）。

## 6. 運用のコツ

- **1バッチ 50〜100通**を目安に。実測規模（案件 約1,850通/日）では、手動での全件処理は現実的でないため、多い日は複数バッチに分けます。
- **①→②→（必要なら③）を1セッションで通す。** 途中で止めるとロックが残ります。
- **複数人で回すのは問題ありません。** ただし同時に取込を始めると片方がロック待ちになるので、声を掛け合うとスムーズです。
- **困ったら Claude にそのまま相談**してください（「ロックが残っているみたい、確認して」等）。スクリプトのコマンドを覚える必要はありません。

## 7. 用語ミニ辞典

| 用語 | 意味 |
|---|---|
| **案件** | 技術者を**募集している**現場の情報（求む側）。台帳に貯める対象。 |
| **人材** | エンジニアを**売り込んでいる**情報（提案側）。v0では取込対象外（仕分けはする）。 |
| **要員** | 自社の稼働可能なエンジニア。マッチングの「人」側。 |
| **案件台帳** | 案件を貯める共有スプレッドシートのタブ。 |
| **本人タブ** | 要員ごとのマッチ結果を貯めるタブ（タブ名＝要員名）。 |
| **messageId** | Gmail API のメール固有ID。重複回避の鍵。 |
| **SES-AI/processed** | 「見た（処理済み）」を表す Gmail ラベル。付くと次回取得の対象外に。 |
| **セッションロック** | 取込中に他の人の取込をブロックする仕組み。append 完了で解放。 |
| **triage / structure / match** | 仕分け / 構造化 / マッチング（3スキルに対応）。 |
