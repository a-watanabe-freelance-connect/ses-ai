# 仕様書: /ses-triage — 案件/人材メールの仕分けスキル

- 作成日: 2026-07-03 ／ 対象: v0
- 根拠: `CLAUDE.md` §実現1（フロー1〜3）・§規模の前提、`DECISIONS.md` §3（SES専用受信箱＝機械フィルタ無し）・§8（3スキル構成）
- 実装先: `.claude/skills/ses-triage/SKILL.md`（**本仕様書が正本**。SKILL.md は本仕様に準拠して作成・更新する）

## 1. 目的・位置づけ

Gmail から取得した未処理メール（`data/inbox/*.jsonl`）を Claude の**推論**で **案件／人材／その他** の3分類に仕分け、結果を `data/triage/*.jsonl` に記録する。v0 パイプラインの第1スキル。

```
fetch_gmail.py → data/inbox → 【/ses-triage 本スキル】 → data/triage
                                                          → /ses-structure（案件のみ構造化→台帳追記→ラベル）
```

- 機械フィルタ（キーワード・正規表現）は**作らない**。SES 専用受信箱＝実質ノイズなしのため、判定はすべて Claude の推論（DECISIONS §3）。
- 本スキルは**ラベルを付与しない**。`SES-AI/processed` ラベルは `/ses-structure` 最終段の `append_sheet.py` が inbox 全件（人材/その他含む）へ一括付与する。仕分けだけで止めてもメールは未処理のまま残る（＝取りこぼし無し）。

## 2. 起動

- 起動例: 「新着メールを取り込んで仕分けして」「inbox を仕分けして」
- オプション（プロンプトで指定可）: 取得上限（`--limit N`）・特定日のみ（`--date YYYY-MM-DD`）・バッチサイズ

## 3. 前提条件

- OAuth 設定済み（`secrets/credentials.json`・`secrets/token.json`）、`.env` 設定済み
- Gmail に `SES-AI/processed` ラベルが存在（無ければ `scripts/setup_v0a.py`）
- Python 仮想環境 `.venv` 構築済み

## 4. 入力（データ契約）

`data/inbox/*.jsonl`（1行1メール）:

```json
{"messageId": "...", "from": "...", "subject": "...", "received_at": "JST ISO8601", "body": "本文テキスト"}
```

`messageId` は Gmail API の message `id`（不変ID・RFC822 Message-ID ではない）。

## 5. 処理手順

1. **取得（必要時）**: 未処理メールを `data/inbox/` へ取得する。
   ```
   .venv/bin/python scripts/fetch_gmail.py [--limit N] [--date YYYY-MM-DD]      # macOS/Linux
   .venv\Scripts\python.exe scripts\fetch_gmail.py [--limit N] [--date YYYY-MM-DD]  # Windows
   ```
   inbox に**未仕分けの jsonl が既にある**場合は fetch を省略しそれを対象にしてよい。
   **cron化フェーズ1（`_state`/`_runlog`）以降、`fetch_gmail.py` はこのバッチの処理サイクル全体を守るセッション単位ロックを取得する**（`/ses-structure` の `append_sheet.py` 完了まで保持）。ロック中は他の人の `fetch_gmail.py` が拒否されるため、triage→structure→append まで通しで進める（放置しない）。
2. **未仕分けファイルの特定**: `data/triage/` に同名ファイルが無い inbox ファイルが対象。
3. **仕分け**: inbox ファイルを **50〜100通/バッチ**で読み、§6 の基準で各メールを 案件/人材/その他 に分類。
4. **出力**: `data/triage/<inboxと同じファイル名>.jsonl` に1行1メールで書き出す（inbox と同件数・messageId の欠落や重複を作らない）。
5. **突合検証（必須）**: `scripts/verify_triage.py` で inbox と triage の messageId 集合が一致することを確認する（未仕分け・重複・欠落を残したまま次段 `/ses-structure` の `append_sheet.py` が実行されるとラベル付与により取りこぼしが発生するため必須。DECISIONS §8）。
   ```
   .venv/bin/python scripts/verify_triage.py [ファイル名（拡張子なし）]      # macOS/Linux
   .venv\Scripts\python.exe scripts\verify_triage.py [ファイル名（拡張子なし）]  # Windows
   ```
   引数省略時は `data/inbox` 全ファイルが対象。messageId 集合の一致に加え、`分類` が `案件`／`人材`／`その他` の3値以外でないか、`理由` が空でないかも検証する。NG が出た場合は該当 triage ファイルを作り直す。
6. **報告**: §8 の形式でサマリを画面表示。

## 6. 判断基準（仕分けルール）

案件と人材は語彙がほぼ同じで、**違いは方向**（求む＝案件／提案＝人材）。

| 分類 | 方向 | 典型シグナル |
|---|---|---|
| **案件** | 現場が技術者を**求めている** | 「要員募集」「〜案件のご紹介」＋必須スキル列挙・単価**上限**（〜80万・MAX80等）・精算幅・面談回数・商流・「ご提案お待ちしております」 |
| **人材** | 技術者を**売り込んでいる** | 「要員のご紹介」「弊社所属エンジニア」・氏名イニシャル・年齢・稼働可能日・希望単価（**下限**）・スキルシート添付への言及 |
| **その他** | どちらの実体も無い | 挨拶・会社案内・セミナー/イベント告知・システム通知・ニュースレター・実案件/実要員の情報が無い定型営業。**件名に「案件」「要員」等のマーカーがあっても、本文に必須スキル/単価/経歴などの実体が無ければ「その他」**（本文が `[skipped: …]` マーカーの配信元も含む） |

### 決め手（優先順位順・件名だけで決めない）

**件名の「〜のご紹介」だけでは判別不能**（案件・人材の両方に出現する。実データでは案件側にも22通）。**必ず本文の「依頼文の向き」で判定する。** 上から順に見て、最初に確定したシグナルで決める:

1. **依頼文の向き（最優先）**: 「下記**案件**に見合う**人材**を探しております」「ご提案お待ちしております」＝**案件**／「下記**人材**にてご提案可能な**案件**ございましたら」「案件があればご紹介ください」＝**人材**。
2. **単価トークンの向き**: `MAX◯` `〜◯万` `上限◯`（＝上限予算）＝**案件**／`@◯万〜` `◯万〜`（＝下限希望）＝**人材**。
3. **主語**: 「弊社プロパー」「弊社所属エンジニア」「イニシャル＋年齢（例: K.K・63歳）」「稼働可能」が主語＝**人材**／「募集」「参画」「必須スキル」が主語＝**案件**。

### 対照 few-shot（同じ「ご紹介」件名でも本文の向きで割れる）

| 件名（例） | 本文の決め手 | 正解 | 理由の書き方 |
|---|---|---|---|
| 【7/7】インフラ設計/大手町/〜100万 のご紹介 | 「必須スキル…下記案件に見合う人材を探しております」＋「金額：〜100万円まで」 | **案件** | 「必須スキルと単価上限(MAX100)を提示し要員を募集しているため案件」 |
| 【即日】DBA/K.K・63歳/仁川駅 のご紹介 | 「個人事業主K.K、DBA歴、稼働可能」＋「単価105万円〜／マッチする案件があれば」 | **人材** | 「個人DBA(K.K)の経歴・希望単価(下限105万〜)を提示し案件を探す方向のため人材」 |
| 弊社プロパー Java SE のご提案 | 「弊社所属エンジニア、@62万〜、スキルシート添付」 | **人材** | 「弊社所属エンジニアの下限単価(@62万〜)とスキルシートを提示する売り込みのため人材」 |

**倒し方（迷ったとき）**:
- 上記1〜3のいずれも確定できず案件か人材か判別困難 → **「案件」に倒す**（次段 `/ses-structure` で案件情報が抽出できなければ落ちる＝安全側）。
- 1通に案件と人材が混在 → **「案件」**とし、理由に「人材情報も含む」と記す（人材取込は将来スコープ）。
- **本文が技術的に読めない（空・文字化け・スタブ数十字・HTML抽出失敗）**場合: いきなり「その他」にせず**まず件名を見る**。件名に明確な方向シグナル（`【当社社員】`『自社要員』＝**人材**／具体的な必須スキル＋単価上限＝**案件**）があれば**件名で判定**し、理由に「本文読取不可・件名から判定（確信度低）」と記す。件名にも手がかりが無ければ **「その他」**（理由「判定不能（本文空/文字化け・件名も手がかりなし）」）。
  - ※ 上表その他の「件名マーカーはあるが本文に実体なし」との違い: あちらは**本文が読めて実体が無い**（＝ノイズ）ケース。ここは**本文が読めない**技術的欠損のケースで、件名の明確シグナルを捨てないための緩和。

## 7. 出力（データ契約）

`data/triage/*.jsonl`（ファイル名は対応する inbox と同名。1行1メール）:

```json
{"messageId": "...", "分類": "案件", "理由": "必須スキルと単価上限を提示して要員を募集しているため"}
```

- `分類` は `案件`／`人材`／`その他` の3値のみ。
- `理由` は1文（次段や後日の検証で使う）。
- **制約**: 対応する inbox ファイルの全 messageId を網羅すること（欠落・重複禁止）。バッチ途中で中断する場合も**ファイル単位で完結**させる（中途半端な triage ファイルを残さない。残った場合は当該ファイルを作り直す）。

## 8. 完了報告（画面表示）

- 冒頭に件数サマリ: 合計 X 通＝案件 A ／人材 B ／その他 C
- 「その他」の内訳例を数件（誤仕分けの点検用）
- 未仕分けの inbox ファイルが残っている場合はその旨と続きの実行案内
- 次アクションの案内: 「案件 A 通を `/ses-structure` で構造化・台帳追記できます」

## 9. エラー・例外

- inbox が空／fetch 結果 0 件 → 「未処理メールなし」と報告して終了。
- fetch_gmail.py の失敗（認証切れ等）→ External+Testing の refresh token は約7日で失効しうる。`secrets/token.json` を削除して再同意（CLAUDE.md §v0実装契約）。
- **`[abort] ロック取得に失敗しました`**（`_state.lock_owner` に他の人が保持中）→ 他の人が `/ses-structure` まで完了させるのを待つか、`_state` タブの `lock_owner`/`lock_acquired_at` を確認のうえ放置と分かれば `python scripts/state.py lock release --force` で手動解除する（TTL既定6時間で自然解放もされる）。
- **`[abort] 本日の daily_budget を使い切っている`**（`_state.daily_budget` 到達）→ `python scripts/state.py budget status` で `day_spent`/`daily_budget` を確認。異常な消費でなければ翌日まで待つか `_state.daily_budget` を人間が調整する。
- **`kill_switch=on のため中断しました`** → 人間が意図的に全処理を止めている状態。`python scripts/state.py killswitch off` で解除してから再実行する。
